Horizontal vs Vertical Scaling — Tradeoffs

CoreHigh Frequency
▾
💡What Is It?

Scaling means making your system handle more users or traffic. There are two ways to do it:

🍔 Analogy — The Restaurant

Imagine you run a burger restaurant. Your single chef can handle 50 orders/hour. Demand doubles.

Vertical Scaling = Hire a superhero chef who works twice as fast. One person, more power.

Horizontal Scaling = Hire 2 normal chefs. More people, same power each.

  • Vertical Scaling (Scale Up): Add more CPU, RAM, or storage to your existing single server. Bigger machine = more capacity.
  • Horizontal Scaling (Scale Out): Add more servers. Instead of one powerful machine, you have many average machines working together.
🔥Why Do We Need It?

No single machine can handle millions of concurrent users. When Twitter gets 500M tweets/day, or Netflix streams to 200M+ users simultaneously, one server would instantly collapse. Scaling is what separates a toy project from a production system.

⚠️ Without Scaling

Your server maxes out at ~10,000 users. At 10,001 users, response times shoot to 30+ seconds. At 50,000 users, complete crash. Black Friday kills your business.

🏢Real-World Example — Netflix vs Early YouTube

YouTube (early days): Started with one big server (vertical scaling). As users grew from thousands to millions, they kept buying bigger and bigger machines. Hit a wall — no single machine can handle 500 hours of video uploaded per minute.

Netflix today: Runs on thousands of AWS instances (horizontal scaling). During peak hours (8–10pm), they can spin up hundreds of new servers in minutes using auto-scaling groups. They never hit a wall because they just add more nodes.

⚙️How It Works Internally
VERTICAL SCALING (Scale Up):
─────────────────────────────
Before:  [Server: 4 CPU, 16GB RAM]  → handles 1,000 req/s
After:   [Server: 32 CPU, 256GB RAM] → handles 8,000 req/s
Limit:   Hardware ceiling exists (biggest machine ~128 CPU)
Risk:    Single point of failure — if this dies, EVERYTHING dies

HORIZONTAL SCALING (Scale Out):
─────────────────────────────────
Before:  [Server 1]  → handles 1,000 req/s

After:   [Server 1] ─┐
         [Server 2] ─┤─── [Load Balancer] ──▶ Users
         [Server 3] ─┘
         → handles 3,000+ req/s
         
Add more:  [Server 1..N] → theoretically unlimited scale
Requires:  Load Balancer + Stateless design
⚖️Advantages & Disadvantages

✅ Vertical Pros

  • Simple — no code changes needed
  • Low latency — no network hops between components
  • Works for stateful apps easily
  • Quick to implement initially

❌ Vertical Cons

  • Hardware ceiling — can't scale forever
  • Single point of failure
  • Expensive at high end
  • Downtime required for upgrades

✅ Horizontal Pros

  • Theoretically unlimited scale
  • No single point of failure
  • Cost-effective (commodity hardware)
  • Zero-downtime deployments possible

❌ Horizontal Cons

  • Complex — need load balancer, distributed state
  • Consistency challenges across nodes
  • Higher network latency
  • Code must be designed for it

Bottlenecks in Scaling

CoreHigh Frequency
▾
💡What Is It?
🚗 Analogy — The Highway Tollbooth

Imagine a 10-lane highway that narrows to 1 tollbooth. You can add more lanes upstream, but everything gets stuck at that one booth. That tollbooth is your bottleneck — the slowest part of the system that limits overall throughput.

A bottleneck is any component in your system that becomes the limiting factor for performance. No matter how much you scale other parts, performance won't improve until the bottleneck is fixed.

🔍Common Bottlenecks
System Request Flow:
User → DNS → Load Balancer → App Server → Cache → Database → Disk

Common Bottleneck Locations:
┌─────────────────────────────────────────────────┐
│  1. DATABASE   ← Most common. N+1 queries,      │
│                  no indexes, too many writes     │
│                                                 │
│  2. NETWORK    ← Bandwidth saturated, high      │
│                  latency between services        │
│                                                 │
│  3. DISK I/O   ← HDD instead of SSD,           │
│                  too many disk reads            │
│                                                 │
│  4. CPU        ← Unoptimized algorithms,        │
│                  crypto operations, encoding     │
│                                                 │
│  5. MEMORY     ← Memory leaks, no caching,      │
│                  large object allocations        │
│                                                 │
│  6. LOCKS      ← Contention in DB transactions, │
│                  distributed locks               │
└─────────────────────────────────────────────────┘
🏢Real-World Example — Twitter's "Database at the Bottom"

In 2008, Twitter was famous for its "Fail Whale" page. Their bottleneck was MySQL. Every tweet fan-out (delivering a tweet to all followers) required N database writes — one per follower. A celebrity with 1M followers sending one tweet caused 1 million DB writes instantly. The database was the tollbooth.

Solution: They built a pre-computed timeline cache (Redis). Now writes go to a queue, not directly to DB. DB bottleneck removed.

⚙️How to Identify & Fix Bottlenecks
STEP 1: Measure (USE Method)
  U = Utilization  (is CPU at 95%? DB at 100%?)
  S = Saturation   (is there a queue building up?)
  E = Errors       (timeout errors? connection refused?)

STEP 2: Profile the Request Path
  User Request
    → 2ms  DNS lookup        ✅ fine
    → 5ms  Load Balancer     ✅ fine  
    → 15ms App Server logic  ✅ fine
    → 890ms Database query   ❌ BOTTLENECK HERE!

STEP 3: Fix Strategies
  DB Bottleneck:  → Add indexes, query optimization, caching, read replicas
  CPU Bottleneck: → Horizontal scale, optimize algorithm, offload to async
  Memory:         → Caching strategy, reduce object size, pagination
  Network:        → CDN, compression, connection pooling, co-location
  Disk I/O:       → SSD, memory-mapped files, async I/O, caching
🎤Interview Perspective
💡 Interview Gold — Systematic Bottleneck Analysis

In system design interviews, after designing your system, proactively say: "Let me identify potential bottlenecks..." This shows senior-level thinking. Always trace the critical path and identify the slowest component.

Common mistake: Candidates add more servers without asking "Where is the actual bottleneck?" More web servers don't help if the DB is the limit.

Stateless Application Design

CorePattern
▾
💡What Is It?
📞 Analogy — The Call Center

Stateless = A call center where ANY agent can handle your call. You always have to identify yourself at the start. No single agent "remembers" you.

Stateful = A call center where you MUST talk to the same agent who started your case, because they hold all the notes.

A stateless application is one where each request contains all the information needed to process it. The server doesn't store any client-specific data between requests. Every request is self-contained.

⚙️Architecture
STATEFUL (Problem):
  User → [Server A] → stores session in Server A's memory
  Next request: User → [Server B] → "Who are you? I have no session!"
                                     ❌ Login failed!

STATELESS (Solution):
  All state (session) stored externally:
  
  User Request ──────────────────────────────────┐
  (contains JWT token or session ID)             │
         │                                       ▼
  [Load Balancer]                         [Redis Cache]
    /   |   \                              (Sessions)
  [S1][S2][S3]  ← any server can handle!
    \   |   /        │
     \  |  /         └── All servers read/write same store
      \ | /
  [Shared Database]

The key insight: servers become interchangeable. Any server can handle any request. State lives in a shared store (Redis, DB), not on the server itself.

🏢Real-World Example — Uber's Ride Matching

When you request an Uber ride, your request hits one of thousands of identical app servers. The server doesn't "know" you from before. Your JWT token carries your identity. Your location, preferences, and ride state are stored in a centralized system (Cassandra + Redis). Any server can pick up your request at any time — even mid-ride if a server dies.

⚖️Pros & Cons

✅ Advantages

  • Easily horizontally scalable — add servers freely
  • Any server can handle any request
  • Servers can die/restart without data loss
  • Zero-downtime deployments easy
  • Simpler load balancing (no sticky sessions)

❌ Disadvantages

  • Every request must carry authentication token
  • External state store (Redis) becomes a dependency
  • Slightly more network overhead per request
  • Shared state store can itself become a bottleneck

Stateful Application Design

PatternCore
▾
💡What Is It?
🎮 Analogy — Online Gaming Session

In an online game, your game server must track exactly WHERE you are, your inventory, health, active enemies. You can't just switch servers mid-game — your entire game state lives on that one server. That's a stateful service.

A stateful application remembers previous interactions. The server stores client-specific data (session, context, connection state) locally and depends on it for future requests from the same client.

⚙️How Stateful Systems Work
Stateful Architecture:
─────────────────────
Client A ──────────────────▶ [Server 1] 
                              └── Memory: {userA: session, cart, context}
                              MUST always talk to Server 1

Client B ──────────────────▶ [Server 2]
                              └── Memory: {userB: session, cart, context}
                              MUST always talk to Server 2

Load Balancer uses STICKY SESSIONS (IP hash / session cookie)
to ensure Client A always reaches Server 1.

Problem: If Server 1 dies → Client A loses ALL state ❌
Problem: Server 1 is 95% full, Server 2 is 10% — can't rebalance ❌
🏢Real-World Examples
Netflix WhatsApp Google Docs
  • WebSocket connections (WhatsApp/Slack): A persistent TCP connection is maintained between your phone and a specific server. The server holds your connection state.
  • Google Docs real-time editing: The operational transformation state for a document lives on one server. All collaborators connect to the same node.
  • Online gaming servers: Your game state (position, health, inventory) lives in-memory on the game server for ultra-low latency.
⚖️Pros & Cons

✅ Advantages

  • Extremely low latency — no external DB lookup
  • Rich context — server knows full history
  • Real-time features natural (WebSockets)
  • Supports streaming & long-lived connections

❌ Disadvantages

  • Hard to scale — sticky sessions required
  • Server failure = state loss (unless replicated)
  • Uneven load distribution possible
  • Complex deployment — can't just swap servers

Load Balancing

CoreHigh FrequencyInfrastructure
▾
💡What Is It?
🏦 Analogy — Bank Teller Queues

You walk into a bank with 5 tellers. A manager (load balancer) at the door tells each customer which teller to go to, ensuring no single teller has a massive queue while others sit idle. If Teller 3 takes a break, the manager stops sending customers there.

A load balancer is a component that distributes incoming network traffic across multiple backend servers. It ensures no single server is overwhelmed, maximizes availability, and increases throughput.

⚙️Architecture & Components
                     ┌─────────────────────┐
Internet Traffic     │    LOAD BALANCER     │    Health checks
────────────────────▶│  (AWS ALB / Nginx /  │◄───every 30s to
Millions of          │   HAProxy / F5)      │    each backend
requests/second      └──────────┬──────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
         [Server 1]        [Server 2]        [Server 3]
         CPU: 45%          CPU: 52%          CPU: 38%
         ✅ Healthy         ✅ Healthy         ✅ Healthy
              │                 │                 │
              └─────────────────┼─────────────────┘
                                ▼
                          [Database Tier]

Load Balancer responsibilities:
  1. Route traffic (using algorithm: round-robin, least-conn, etc.)
  2. Health check backends — remove dead servers from pool
  3. SSL termination — decrypt HTTPS, forward HTTP internally
  4. Session persistence (sticky sessions if needed)
  5. Rate limiting (some LBs)
  6. DDoS protection
🏢Real-World Example — Amazon's Load Balancers

Amazon.com handles Black Friday with Application Load Balancers (ALBs) in front of thousands of microservices. Each service (cart, search, payments) has its own load balancer. The ALB inspects HTTP headers/URLs to route /api/cart/* to Cart Service instances and /api/search/* to Search Service instances. If any instance fails its health check (HTTP 200 not returned within 2s), the ALB automatically removes it from rotation within seconds.

⚖️Pros & Cons

✅ Advantages

  • High availability — automatically routes around failures
  • Scalability — add servers without downtime
  • SSL termination — offload crypto from app servers
  • Security — hides backend topology
  • Zero-downtime deployments (rolling updates)

❌ Disadvantages

  • Load balancer itself can become a SPOF (need redundant LBs)
  • Adds latency (small, typically <1ms)
  • Stateful apps need sticky sessions
  • Additional cost and complexity

Layer 4 Routing (Transport Layer)

InfrastructureCore
▾
💡What Is It?
📮 Analogy — Postal Sorting by ZIP Code

L4 routing is like a postal sorting machine that only looks at the ZIP code (IP address + port) on the envelope — it doesn't open the envelope to read the letter inside. It routes based purely on destination address, not content.

Layer 4 refers to the Transport Layer of the OSI model (TCP/UDP). An L4 load balancer routes traffic based on IP address and TCP/UDP port number only. It doesn't inspect the content of the packets.

⚙️How L4 Routing Works
OSI Model Context:
  Layer 7 - Application  (HTTP, HTTPS, gRPC)
  Layer 6 - Presentation (SSL/TLS)
  Layer 5 - Session
  Layer 4 - Transport    ← L4 Load Balancer operates here
  Layer 3 - Network      (IP addresses)
  Layer 2 - Data Link
  Layer 1 - Physical

L4 Load Balancer sees:
  Source IP:   192.168.1.5
  Dest IP:     10.0.0.1
  Dest Port:   443 (HTTPS)
  Protocol:    TCP
  Payload:     [NOT INSPECTED - treated as raw bytes]

L4 Routing Decision:
  Client ──TCP SYN──▶ [L4 LB: port 443]
                           │
                     IP hash or Round Robin
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
     [Server 1:443]  [Server 2:443]   [Server 3:443]

TCP connection is FORWARDED (NAT) — LB modifies dest IP
LB itself doesn't terminate the TCP connection (passthrough mode)
Very fast: ~100 microseconds latency overhead
🏢Real-World Use — AWS Network Load Balancer (NLB)

When you're running a game server (UDP) or a financial trading system (ultra-low latency TCP), AWS NLB is an L4 load balancer. It handles millions of requests/second with ~100μs latency because it just forwards TCP packets — no HTTP parsing, no URL inspection. Netflix uses NLB for its control plane traffic.

Layer 7 Routing (Application Layer)

InfrastructureHigh Frequency
▾
💡What Is It?
📦 Analogy — Smart Package Sorter

L7 routing is like a sorter who OPENS each package, reads the label inside, and routes it to the right department. "This says 'video content' — send to the Video Team. This says 'payment' — send to Finance, with extra security." Intelligent content-based routing.

Layer 7 routing operates at the Application layer. An L7 load balancer reads the full HTTP request — headers, URL path, cookies, body — and makes intelligent routing decisions based on this content.

⚙️L7 Routing Capabilities
L7 Load Balancer (Nginx / HAProxy / AWS ALB / Envoy):

Incoming request: POST /api/v2/payments HTTP/1.1
                  Host: api.amazon.com
                  Authorization: Bearer eyJ...
                  X-User-Region: US-WEST
                  Content-Type: application/json

L7 can route based on:
┌──────────────────────────────────────────────────────┐
│  URL Path:    /api/payments/* → Payment Service      │
│  URL Path:    /api/cart/*    → Cart Service          │
│  URL Path:    /api/search/*  → Search Service        │
│                                                      │
│  Host:        api.amazon.com  → API Cluster          │
│  Host:        cdn.amazon.com  → CDN Cluster          │
│                                                      │
│  Header:      X-Version: v2   → New backend (A/B)   │
│  Cookie:      beta=true       → Beta servers         │
│                                                      │
│  Weight:      10% → Canary, 90% → Stable (canary)   │
└──────────────────────────────────────────────────────┘

Path-based routing for microservices:
                    ┌── /api/users   ──▶ [User Service ×3]
                    │
Client ──▶ [L7 LB] ┼── /api/orders  ──▶ [Order Service ×5]
                    │
                    └── /api/search  ──▶ [Search Service ×10]

Round Robin Algorithm

AlgorithmCore
▾
💡What Is It?
🎲 Analogy — Dealing Cards

Round Robin is like a card dealer distributing cards one by one to each player in sequence: Player 1, Player 2, Player 3, Player 1, Player 2, Player 3... Each player gets exactly the same number of cards regardless of how fast they play.

Round Robin is the simplest load balancing algorithm. Requests are distributed to backend servers in sequential order, cycling through the server list evenly.

⚙️How It Works
Server Pool: [S1, S2, S3]
Counter starts at 0

Request 1  → index 0 % 3 = 0 → Server S1
Request 2  → index 1 % 3 = 1 → Server S2
Request 3  → index 2 % 3 = 2 → Server S3
Request 4  → index 3 % 3 = 0 → Server S1
Request 5  → index 4 % 3 = 1 → Server S2
...

Pseudocode:
  servers = [S1, S2, S3]
  counter = 0
  
  def get_server():
      server = servers[counter % len(servers)]
      counter += 1  // atomic increment
      return server

WEIGHTED Round Robin:
  S1 weight=3, S2 weight=1, S3 weight=1
  Distribution: S1,S1,S1,S2,S3,S1,S1,S1,S2,S3...
  (S1 gets 3x more requests — it's a bigger machine)

Visual:
  Time →
  Req: 1  2  3  4  5  6  7  8  9
  LB:  S1 S2 S3 S1 S2 S3 S1 S2 S3
⚖️Pros & Cons

✅ Advantages

  • Extremely simple to implement — O(1) per request
  • No state needed about server health/load
  • Works perfectly when requests are equal cost
  • Predictable distribution

❌ Disadvantages

  • Ignores actual server load — may overload slow servers
  • Not ideal for variable-length requests
  • Doesn't account for server capacity differences
  • Long-running connections cause imbalance
⚠️ When Round Robin Fails

If Request 1 takes 500ms (a slow DB query) and Request 2 takes 1ms, Round Robin treats them identically. Server 1 gets stuck while Server 2 is idle. Least Connections would solve this better.

Least Connections Algorithm

AlgorithmCore
▾
💡What Is It?
🏥 Analogy — Hospital Emergency Room

In an ER, a triage nurse looks at all the doctors and sends the next patient to whichever doctor currently has the fewest patients. Not "whose turn is it" but "who is least busy right now." This is Least Connections.

The Least Connections algorithm routes each new request to the backend server with the fewest active connections at that moment. It dynamically adapts to actual server load rather than just rotating through the list.

⚙️How It Works
Server State Tracking:
  S1: 45 active connections
  S2: 12 active connections  ← MINIMUM
  S3: 28 active connections

Next request → route to S2 (fewest connections)

After routing to S2:
  S1: 45 connections
  S2: 13 connections  ← updated
  S3: 28 connections

Pseudocode:
  def get_server(servers):
      return min(servers, key=lambda s: s.active_connections)

WEIGHTED Least Connections:
  Normalize by capacity: score = connections / weight
  S1: 45/4 = 11.25  (powerful 4x server)
  S2: 12/1 = 12.00
  S3: 28/2 = 14.00  ← route here (best normalized)

When to use Least Connections:
  ✅ Long-lived connections (WebSockets, database connections)
  ✅ Variable request processing time
  ✅ Heterogeneous workloads (mix of fast/slow requests)
  ❌ Not needed for uniform short requests (Round Robin is fine)
🏢Real-World Use — Database Connection Pooling

When PgBouncer (PostgreSQL connection pooler) distributes application requests across DB read replicas, it uses Least Connections. Some queries take 200ms, some take 2 seconds. Without Least Connections, Round Robin would pile 10 slow queries on one replica while another is idle. With LC, traffic naturally flows to the less-busy replica.

⚖️Pros & Cons

✅ Advantages

  • Adapts to actual server load dynamically
  • Prevents server overload for long requests
  • Better for variable-length workloads
  • Self-balancing as connections complete

❌ Disadvantages

  • Requires tracking connection count (stateful LB)
  • Overhead: min-search across all servers per request
  • Connection count ≠ CPU load (a query can be CPU-heavy with 1 conn)
  • Race condition: two LB instances may pick same server simultaneously

Consistent Hashing Algorithm

AlgorithmHigh FrequencyMust Know
▾
💡What Is It?
🎡 Analogy — The Circular Seating Chart

Imagine a circular table with seats numbered 0–360. Guests (requests) are assigned a number. Servers are also assigned a number. Each guest sits at the nearest server seat going clockwise. If a server leaves, only their guests shift to the next server — everyone else stays put. Adding a server only disturbs guests near that seat, not the entire table.

Consistent Hashing is an algorithm used in distributed systems to evenly distribute data across nodes (servers/caches) while minimizing data reorganization when nodes are added or removed.

Without it: adding 1 server to a 10-server pool reshuffles ~91% of keys. With consistent hashing: only ~1/n (10%) of keys need to move.

⚙️How Consistent Hashing Works
STEP 1: Create a hash ring (0 to 2^32)
  
        0 ──────────────────────── 2^32
        │          HASH RING         │
        └────────────────────────────┘
                (circular)

STEP 2: Map servers to ring positions using hash(server_id)

        0°
       ┌─────┐
  270° │     │ 90°   Servers placed on ring:
       │  ●──────── Server A: hash("A") = 45°
       │     │      Server B: hash("B") = 130°
  180° └─────┘      Server C: hash("C") = 220°
                    Server D: hash("D") = 310°

STEP 3: For each key, hash(key) → find next server clockwise

  hash("user:123") = 100° → Server B (next clockwise from 100°)
  hash("user:456") = 180° → Server C (next clockwise from 180°)
  hash("user:789") = 280° → Server D

STEP 4: Add Server E at 170° → ONLY keys between 130°-170° move
  Old: keys 130°-220° → Server C
  New: keys 130°-170° → Server E
       keys 170°-220° → Server C (unchanged)
  Only ~25% of Server C's keys moved! ✅

VIRTUAL NODES (to fix uneven distribution):
  Each server gets 150 virtual nodes spread across ring
  Server A: hash("A#1"), hash("A#2"), ..., hash("A#150")
  This ensures even distribution even with few physical servers
🏢Real-World Usage
Amazon DynamoDB Facebook Cassandra Amazon ElastiCache Twitter
  • Amazon DynamoDB: Uses consistent hashing to shard data across partition nodes. Adding capacity only moves ~1/n of the data.
  • Apache Cassandra: Each node owns a range on the hash ring. Data is replicated to N successor nodes for fault tolerance.
  • Memcached/Redis Cluster: Client-side consistent hashing to route GET user:123 always to the same cache node — maximizing cache hit rate.
  • CDN Edge Selection: Akamai uses consistent hashing to route requests for the same URL to the same edge server, improving cache efficiency.
⚖️Pros & Cons

✅ Advantages

  • Only ~1/n keys remapped when adding/removing nodes
  • Horizontally scalable — add nodes with minimal disruption
  • Same key always hits same node (cache locality)
  • Works in distributed systems without central coordinator

❌ Disadvantages

  • Non-uniform distribution without virtual nodes
  • Hot spots possible if data is not uniformly distributed
  • More complex to implement than simple modulo hashing
  • Virtual node management adds overhead
🔍Why Not Simple Modulo Hashing?
PROBLEM with simple hashing: server = hash(key) % N

  With N=3 servers:
    "user:1" → hash % 3 = 0 → Server 0
    "user:2" → hash % 3 = 1 → Server 1
    "user:3" → hash % 3 = 2 → Server 2

  Add 1 server (N=4):
    "user:1" → hash % 4 = 3 → Server 3  ❌ MOVED!
    "user:2" → hash % 4 = 2 → Server 2  ❌ MOVED!
    "user:3" → hash % 4 = 3 → Server 3  ❌ MOVED!
  
  ~75% of all keys get remapped! 
  Cache miss storm! Database gets hammered!

CONSISTENT HASHING solution:
  Adding 1 server to 10-node ring → only ~10% keys remapped ✅

💡What Is It?

Walmart doesn't ship everything from a single warehouse in Arkansas. They have stores in every city so products are near customers. A CDN is a globally distributed network of servers (called PoPs — Points of Presence) that cache and serve content from the location nearest to each user. Instead of everyone hitting your origin server in Virginia, users in Sydney hit a CDN server in Sydney.

  User in London ──▶ [CDN London PoP] ──(cache hit)──▶ User

  CDN Sydney ──(cache miss, first time)──▶ [Origin: Virginia]

Push Caching Strategy

PatternCore

💡What Is It?
🛒 Analogy — Grocery Store Restocking

Push caching is like a store manager who pre-stocks shelves every morning based on what he KNOWS will sell that day (milk, bread, eggs). He doesn't wait for customers to ask — he proactively fills the shelves before demand arrives. Content is "pushed" to the cache before it's requested.

In Push caching, the origin server or your application proactively pushes content to the cache (CDN edge nodes). You decide what goes into the cache and when, rather than letting the cache decide based on incoming requests.

⚙️How Push Caching Works
Push Flow:
  1. Content is created/updated on origin
  2. Origin proactively uploads to CDN edge nodes
  3. Cache is warm BEFORE any user requests
  4. User requests are served instantly from warm cache

  [New Movie Published] ──(push)──▶ [CDN Node: Tokyo  ]
         │                          [CDN Node: London  ]
         │               ──(push)──▶[CDN Node: NY      ]
         │                          [CDN Node: Sydney  ]
         └── All nodes seeded before movie goes live!

Tools for Push Caching:
  CDN Push:     Manually upload to CDN origin bucket (S3 + CloudFront)
  Redis SET:    Proactively write to Redis cache after DB write
  Memcached:    Pre-populate after ETL jobs complete
  CDN APIs:     PUT /cache/{key} via CDN API

When to Use Push:
  ✅ Known in advance: product catalog, static pages, scheduled content
  ✅ High-traffic items: homepage, hero images, top 100 products
  ✅ After cache clear: repopulate critical content immediately
  ✅ Scheduled events: prime cache for product launches, sports events
  
When NOT to Use Push:
  ❌ Unpredictable content (what will go viral?)
  ❌ User-specific content (100M variations)
  ❌ Very large content sets (can't push everything)
🏢Real-World — Netflix Pre-Positioning

Netflix calls their push caching strategy "pre-positioning". Every night between 2-6 AM (low traffic), Netflix's algorithms predict what content will be popular tomorrow (based on viewing patterns, new releases, geography). They then pre-push those video files to Open Connect Appliances (edge servers) near users. By morning, the cache is warm. Result: near-zero cache misses for popular content.

Pull Caching Strategy

PatternCore
▾
💡What Is It?
📚 Analogy — Library Book Requests

Pull caching is like a library that only orders books AFTER a patron requests them. The first person to ask for "Book X" waits while the library orders it. But every person AFTER that gets it instantly from the shelf. The library self-organizes based on actual demand — popular books always on the shelf, rare books only ordered when needed.

In Pull caching, the cache fetches content from the origin on the first request (cache miss), stores it, and serves it from cache for subsequent requests until TTL expires. The cache "pulls" data on demand.

⚙️How Pull Caching Works
Pull Caching Flow:
  
FIRST REQUEST (Cache Miss):
  User ──▶ [CDN/Cache] ──(miss: not cached)──▶ [Origin]
                                                    │
           [CDN/Cache] ◄──(response + cache)────────┘
              │
  User ◄──────┘  (serves user, stores for next time)
  Latency: full origin round trip

SUBSEQUENT REQUESTS (Cache Hit):
  User ──▶ [CDN/Cache] ──(HIT! serve from cache)──▶ User
  Latency: edge server only, <10ms ✅

Cache TTL (Time-To-Live):
  Cache-Control: max-age=3600  → Cache for 1 hour
  After 1 hour, next request is a cache miss → re-fetch

Self-organizing behavior:
  Popular content: frequently requested → always in cache
  Rare content: evicted by LRU → only fetched when needed

CDN Pull Zone (for static sites):
  CloudFront origin = S3 bucket
  First request to /image.jpg → CloudFront pulls from S3
  Subsequent requests → served from CloudFront edge cache
🏢Real-World Example — GitHub Pages + CDN

GitHub Pages uses pull caching via Fastly CDN. When a new Jekyll blog post is published, it's pushed to GitHub's origin. When a user in Tokyo first visits the post, Fastly's Tokyo PoP pulls it from GitHub's origin and caches it (cache miss, ~200ms). The next 10,000 Tokyo visitors get it from Fastly's local cache (<5ms). The CDN self-populates based on actual demand.

🔍Caching Patterns Within Pull
  • Cache-Aside (Lazy Loading): App checks cache → miss → load from DB → store in cache → return
  • Read-Through: Cache sits in front of DB, automatically fetches on miss
  • Write-Through: On write, update both DB and cache simultaneously (ensures freshness)
  • Write-Behind (Write-Back): Write to cache first, async update DB later (risk of data loss)
Cache-Aside Pattern (most common):
  get_user(id):
    user = cache.get("user:" + id)    // Check cache
    if user is None:                   // Cache miss
        user = db.query("SELECT...")   // Hit database
        cache.set("user:"+id, user,    // Store in cache
                  ttl=300)             // 5 min TTL
    return user

  Multiple servers announce SAME IP globally
  Internet routing (BGP) naturally delivers traffic

Rate Limiting & Throttling

CoreHigh FrequencyPattern
▾
💡What Is It?
🚗 Analogy — Highway Speed Limit

A highway can handle 2,000 cars/hour safely. Without a speed limit, during rush hour, 5,000 cars try to enter → gridlock, accidents, ZERO throughput. The speed limit + on-ramp meters are rate limiters — they control the INPUT rate to keep the system flowing smoothly.

Rate limiting controls how many requests a user/client can make in a given time window. Throttling is the enforcement mechanism — slowing down or rejecting requests that exceed the limit.

🔥Why Rate Limiting?
  • DDoS Protection: Attacker can't overwhelm your API with 10M requests/second
  • Fairness: One heavy API consumer can't starve all other users
  • Cost Control: Expensive operations (AI inference, DB writes) capped per user
  • Business Model: Free tier = 100 req/min, Paid tier = 10,000 req/min
  • Prevent Cascading Failure: Downstream service protected from upstream bursts
⚙️Rate Limiting Dimensions
Limit BY what?

  Per API Key:   key_abc123 → max 1000 req/min

  Per Endpoint:  POST /api/ai → 10 req/min (expensive)
                 GET /api/data → 1000 req/min (cheap)
  Per Tenant:    org_enterprise → 10K req/min
  Global:        All traffic → max 1M req/min total

Response when exceeded:
  HTTP 429 Too Many Requests
  Headers:
    Retry-After: 60             → wait 60 seconds
    X-RateLimit-Limit: 100      → your limit
    X-RateLimit-Remaining: 0    → remaining this window
    X-RateLimit-Reset: 1700000  → when window resets

Rate Limiting vs Throttling:
  Rate Limiting:  Binary — allow OR reject (429)
  Throttling:     Gradual — slow responses, queuing
  
  Rate Limiting: "You've used 100 requests, wait 60s"
  Throttling:    "Your 101st request will take 5s instead of 5ms"

Token Bucket Algorithm

AlgorithmMust Know
▾
💡What Is It?
🎟️ Analogy — Amusement Park Tickets

You get a bucket that holds 10 tickets. Every minute, 2 new tickets are added (up to the max of 10). Each ride costs 1 ticket. You can use all 10 tickets in a burst (10 rides quickly!) but then you must wait for the bucket to refill. You can't use tickets you don't have.

The Token Bucket algorithm allows bursting up to the bucket capacity while enforcing an average rate equal to the refill rate. It's the most widely used rate limiting algorithm (AWS, Stripe, GitHub all use it).

⚙️How Token Bucket Works
Parameters:
  Bucket capacity: 10 tokens  (max burst size)
  Refill rate:     2 tokens/second  (steady-state rate)
  
Algorithm:
  1. Tokens accumulate at refill_rate (up to capacity)
  2. Each request consumes 1+ tokens
  3. If tokens available → allow request, consume token
  4. If no tokens → reject (429) or queue request

Timeline:
  t=0s:  Bucket = 10 tokens (full)
  
  t=0s:  5 requests burst → uses 5 tokens, bucket=5
  t=0s:  5 more requests  → uses 5 tokens, bucket=0
  t=0s:  1 more request   → REJECTED (0 tokens) ❌
  
  t=1s:  2 tokens added   → bucket=2
  t=1s:  2 requests       → uses 2 tokens, bucket=0

Redis Implementation:
  FUNCTION allow_request(user_id):
    key = "rate:" + user_id
    now = current_timestamp()
    
    // Lua script (atomic):
    tokens = GET key.tokens
    last_refill = GET key.last_refill
    elapsed = now - last_refill
    new_tokens = min(capacity, tokens + elapsed * rate)
    
    IF new_tokens >= 1:
        SET key.tokens = new_tokens - 1
        SET key.last_refill = now
        RETURN ALLOW
    ELSE:
        RETURN REJECT
⚖️Pros & Cons

✅ Advantages

  • Allows bursting — good for bursty traffic patterns
  • Smooth average rate control
  • Memory efficient — O(1) per user (just store tokens + timestamp)
  • Most widely used — AWS, Stripe, GitHub use it

❌ Disadvantages

  • Two parameters to tune (capacity + rate)
  • Burst at start of window can still spike backend
  • Race conditions in distributed implementations
  • Slightly complex to implement atomically

Leaky Bucket Algorithm

AlgorithmCore
▾
💡What Is It?
🪣 Analogy — A Real Leaky Bucket

Water (requests) pours into a bucket at any rate (bursts allowed). The bucket has a small hole at the bottom that lets water drip out at a CONSTANT rate. No matter how fast you pour, the output is always the same steady drip. If the bucket overflows (too many requests queued) → excess water (requests) are discarded.

The Leaky Bucket algorithm processes requests at a constant, fixed output rate. Incoming requests go into a queue (bucket). If the queue is full, new requests are dropped. The queue drains at a steady rate — smoothing out bursts.

⚙️How Leaky Bucket Works
Leaky Bucket Parameters:
  Bucket size: 10 (queue capacity)
  Leak rate:   2 requests/second (constant output)

Behavior:
  Input:  Bursty (100 requests in 1 second)
  Output: Smooth (always 2 requests/second)

  t=0s: 10 requests arrive → queue fills to 10
  t=0s: 5 more arrive → DROPPED (queue full) ❌
  
  t=0.5s: 1 request processed and sent to backend
  t=1.0s: 1 request processed and sent to backend
  t=1.5s: 1 request processed and sent to backend
  ...always at 2/sec, regardless of input burst

Visual:
  Input:   ████████████████ (burst)
  Queue:   ████████ (capped)
  Output:  ██ ██ ██ ██ ██  (smooth, constant)

Token Bucket vs Leaky Bucket:
  Token Bucket: Output CAN burst (up to bucket size)
  Leaky Bucket: Output is ALWAYS constant (no burst)
  
  Token Bucket: Better for APIs (allow burst for UX)
  Leaky Bucket: Better for network traffic shaping
🏢Real-World Use — Network Traffic Shaping

Internet Service Providers (ISPs) use leaky bucket to implement QoS (Quality of Service) traffic shaping. If you're on a "10 Mbps" plan, they use a leaky bucket to ensure you never exceed that rate, no matter how fast your computer tries to download. Router chips implement this in hardware (FPGA/ASIC) at wire speed.

In software: Nginx's limit_req directive uses a leaky bucket variant to smooth out request rates.

Sliding Window Log Algorithm

AlgorithmCore
▾
💡What Is It?
📋 Analogy — Moving Clipboard

Imagine a clipboard that shows the last 60 minutes of your activity. At any moment, you count the entries on the clipboard. If there are more than 100, your next request is rejected. As old entries fall off the back (become older than 60 min), new entries can be added. The window "slides" forward in time continuously.

The Sliding Window Log algorithm tracks the exact timestamp of each request in a log. For every new request, it counts how many requests occurred in the past N seconds (the window). If the count exceeds the limit, reject. The window slides with real time — no hard boundaries.

⚙️Sliding Window vs Fixed Window
FIXED WINDOW PROBLEM:
  Window: 1-minute buckets, limit = 100 req/min
  
  12:00:59 → 100 requests (window resets at 12:01:00)
  12:01:01 → 100 more requests ← ALLOWED!
  
  In 2 seconds (12:00:59 to 12:01:01): 200 requests! 
  Double the limit. VULNERABILITY! ❌

SLIDING WINDOW LOG (solves it):
  Always look at EXACTLY the past 60 seconds
  
  At 12:01:01, look back 60s: 12:00:01 to 12:01:01
  That includes the 100 from 12:00:59 ← still counted!
  New request → rejected ✅

Implementation:
  data = SortedSet()  // Redis ZSET: {timestamp: request_id}
  limit = 100
  window = 60 seconds

  FUNCTION allow(user_id, now):
    key = "log:" + user_id
    
    // Remove expired entries (older than window)
    ZREMRANGEBYSCORE key 0 (now - window)
    
    // Count remaining entries in window
    count = ZCARD key
    
    IF count < limit:
        ZADD key now request_id   // Log this request
        EXPIRE key window          // Auto-cleanup
        RETURN ALLOW
    ELSE:
        RETURN REJECT (429)

Memory Usage:
  Stores 1 entry per request per user
  If limit=1000, each user uses ~1000 entries
  HIGH MEMORY: O(requests_per_window * active_users)
  vs Token Bucket: O(1) per user ← much better

30-Minute Interview Cheat Sheet

▼
Pre-Interview Ritual: Read this top-to-bottom 30 minutes before your interview. Each bullet is a "trigger phrase" — if you hear the interview topic, these words should fire in your head.

⚖️ Scaling

  • Vertical = bigger machine → fast but ceiling exists
  • Horizontal = more machines → needs stateless design
  • Databases prefer vertical first, then read replicas
  • App servers prefer horizontal behind a load balancer
  • Magic phrase: "Scale out, not up, for web tier"

🚦 Load Balancing

  • L4 = IP+Port, fast, no content inspection, NLB
  • L7 = HTTP-aware, path routing, ALB, Envoy
  • Round Robin = same hardware, short requests
  • Least Connections = long-lived, mixed load
  • IP Hash = sticky sessions without server-side state
  • LB itself needs HA (Active-Passive or Anycast)

🔁 Stateless vs Stateful

  • Stateless = JWT/token in request, session in Redis
  • Stateful = WebSocket, in-memory, sticky routing needed
  • Prefer stateless → easier to scale, deploy, recover
  • Stateful unavoidable for: real-time games, chat, collaborative edit
  • Magic phrase: "Push state to the edges or external store"

#️⃣ Consistent Hashing

  • Hash ring: keys and nodes on same circle
  • Key goes to first node clockwise
  • Node removed → only its keys remapped (not all)
  • Virtual nodes fix uneven distribution
  • Used by: Cassandra, DynamoDB, Memcached, Redis Cluster

🌍 CDN & Edge

  • CDN = static assets cached at global PoPs
  • Push = pre-load known content (Netflix movies)
  • Pull = cache on first miss (dynamic sites)
  • Edge Servers = run code at PoP (Workers, Lambda@Edge)
  • Cache-Control headers control TTL at CDN
  • Invalidation is hard → use versioned URLs

🌐 Geo-Routing

  • DNS-based = Route 53 Latency / Geolocation
  • Anycast = same IP, BGP routes to nearest PoP
  • GDPR → EU users must stay in EU region
  • Active-Active multi-region → needs conflict resolution
  • Active-Passive → simple but wastes standby capacity

🚧 Rate Limiting

  • Token Bucket = burst allowed, tokens refill at rate R
  • Leaky Bucket = smooth output, queue absorbs bursts
  • Fixed Window = simple but boundary spike problem
  • Sliding Window Log = precise, high memory
  • Sliding Window Counter = approximate, low memory
  • Distributed RL → Redis atomic ops (Lua / INCR+EXPIRE)
  • Return HTTP 429 Too Many Requests + Retry-After

🔍 Bottleneck Hunting

  • USE Method: Utilization, Saturation, Errors per resource
  • Common bottlenecks: DB writes, single cache node, network I/O
  • Fan-out pattern → write to many followers is classic bottleneck
  • Fix toolkit: cache, async queues, read replicas, sharding, CDN
  • Measure first, optimise second — never guess

🚩 Top 10 Mistakes Candidates Make

  1. Jumping to solution without clarifying requirements — Always ask: scale, latency SLA, consistency needs.
  2. Ignoring the database — The DB is almost always the bottleneck. Always mention sharding / read replicas.
  3. Forgetting the Single Point of Failure — Every component you add should be HA or have a failover plan.
  4. Confusing L4 and L7 — L4 = TCP/IP, fast, dumb. L7 = HTTP, smart, slightly slower.
  5. Not mentioning cache invalidation — Interviewers love this. "Cache invalidation is one of two hard problems in CS."
  6. Treating CDN and cache as the same thing — CDN = geographic distribution. Cache = in-memory speed layer.
  7. Not explaining why consistent hashing over modulo — The answer: modulo remaps all keys on node change.
  8. Saying "just use Kafka" without explaining what problem it solves.
  9. Ignoring monitoring/observability — Every design needs metrics, logging, alerting.
  10. Not quantifying scale — "High traffic" means nothing. Say: "10M DAU, 50K req/s, 99.9% SLA."
